# v2.6.1

A patch release fixing OAuth login on WSL. Installing `codex-multi-auth` on a Windows host and inside WSL at the same time broke sign-in in both environments; removing the Windows install was the only thing that made WSL work. Closes [#630](https://github.com/ndycode/codex-multi-auth/issues/630).

Routing, rotation, storage, the account pool, and the token flow are unchanged. Every behavior change is gated behind a new WSL check that is `false` on all other hosts, so native Windows, macOS, and Linux are unaffected.

## Auth

### Fixes

- **WSL logins now open a browser.** WSL reports `process.platform === "linux"`, so the browser launcher fell through to `xdg-open` — which is not installed on a stock WSL Debian. The launcher returned `false` silently, no browser ever opened, and sign-in looked like a hang. Under WSL the Windows browser is now opened via `wslview` (from `wslu`), falling back to `powershell.exe` interop, and only then to the Linux opener ([#630](https://github.com/ndycode/codex-multi-auth/issues/630)).
- **The clipboard now targets Windows under WSL.** The manual-paste fallback is exactly where WSL users were landing, and it was copying to the distro's clipboard rather than the one the user pastes from. Copy now routes through `clip.exe`, then PowerShell `Set-Clipboard`, before the Linux clipboard tools.
- **A callback failure now explains itself instead of silently degrading.** The OAuth redirect URI is registered with the provider as `http://localhost:1455/auth/callback`, so the callback port is fixed and cannot be renegotiated when it is contended. A browser launched from WSL still runs on the Windows host, and Windows resolves `localhost:1455` against its own loopback first — so a Windows-side login or proxy holding that port receives the redirect the WSL listener is waiting for. From inside the distro this is invisible: the listener binds cleanly and simply never sees the callback. `login` now names the conflict, shows how to find the listener on each side, and points at `login --device-auth`, which needs no callback port.
- **A `child.stdin` `error` handler was added to every clipboard path.** A child process that dies before draining stdin emits `EPIPE` on the *stream*, not on the child, and the surrounding `try/catch` cannot catch it. This also hardens the pre-existing `pbcopy` / `xclip` / `xsel` paths.

### Notes on the diagnosis

Contention is only *asserted* when it is actually observed — a listen that fails with `EADDRINUSE`. A callback that never arrives is far more often a cancelled or abandoned sign-in than a stolen redirect, so that case leads with "if you closed or cancelled the browser sign-in, just run login again" and offers the contention explanation conditionally. Inspection commands are platform-correct (`lsof` on macOS, `ss` on Linux, `Get-NetTCPConnection` on Windows; both sides inside WSL).

## Docs

### Features

- New [Windows and WSL side by side](../troubleshooting.md#windows-and-wsl-side-by-side) section in the troubleshooting guide: why the callback port is fixed, how to find the listener on each side of the boundary, and the `--device-auth` escape hatch. Linked from the README troubleshooting list.
- Documented that Windows and WSL keep **separate state directories**, so accounts are not shared between them and each environment must be signed in independently.

## Testing

### Improvements

- 31 new tests: WSL detection (`test/wsl.test.ts`), the failure guidance (`test/callback-guidance.test.ts`), the real `EADDRINUSE` bind path (`test/oauth-server-port-conflict.test.ts`, which squats on port 1455 with a bare `net.createServer`), the `runOAuthFlow` wiring that selects the failure reason (`test/login-oauth-callback-guidance.test.ts`), and a WSL block in `test/browser.test.ts` covering opener fallback ordering, PowerShell metacharacter escaping, and clipboard routing.
- `test/browser.test.ts` now explicitly clears `WSL_DISTRO_NAME` / `WSL_INTEROP` and stubs `readFileSync`. Without that, its native-host assertions would have rerouted through the Windows interop paths for anyone running the suite *from inside* WSL.

## Notes

- **The callback port cannot be changed.** It is baked into the provider-registered redirect URI, which is why this release is *detect and explain* rather than *avoid*. `codex-multi-auth login --device-auth` remains the only way to sign in on both Windows and WSL without coordinating the port.
- A WSL login that loses the callback still waits out the five-minute callback poll before printing the guidance. Emitting the hint early, while continuing to wait, is a candidate follow-up.
- Upgrade with `npm i -g codex-multi-auth`. No state, config, or on-disk layout changes; nothing to migrate, and no action is required for existing installs.
