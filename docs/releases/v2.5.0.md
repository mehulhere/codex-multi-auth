# v2.5.0

Adds the GPT-5.6 model family (Sol, Terra, Luna) as first-class models, along with the two reasoning tiers it introduces above `xhigh`. This is an additive feature release: GPT-5.6 is opt-in, the legacy `gpt-5` alias still resolves to `gpt-5.5`, and the default model, quota-probe chain, routing, rotation, storage, and auth flow are unchanged.

Model facts were taken from the upstream Codex catalog (`openai/codex` → `codex-rs/models-manager/models.json`) and `codex-rs/core/src/client.rs`, not from the published API documentation. The docs pages contradict each other on whether a `max` tier exists, and the context window they advertise applies to the API surface rather than the ChatGPT Codex backend this wrapper talks to.

## Models

### Features

- Added `gpt-5.6-sol`, `gpt-5.6-terra`, and `gpt-5.6-luna` as first-class models with their own profiles, effort aliases, prompt-family routing, capability metadata, and pricing. Bare `gpt-5.6` resolves to Sol, matching OpenAI's own flagship alias.
- Added two reasoning tiers above `xhigh`: `max`, supported by all three tiers, and `ultra`, supported by Sol and Terra only.
- Per-tier defaults mirror the upstream catalog exactly: Sol defaults to `low`, Terra and Luna to `medium`. Sol's unusually low default is intentional and matches `default_reasoning_level` upstream.
- No GPT-5.6 tier accepts `none` or `minimal` reasoning effort. Those requests are coerced up to a supported effort rather than being sent and rejected.
- The three tiers ship in `config/codex-modern.json` with a 372,000-token context window, per the Codex catalog.

### `ultra` never reaches the wire

- `ultra` denotes Codex's client-side automatic subagent delegation, not a distinct backend effort level. Upstream `reasoning_effort_for_request` (`codex-rs/core/src/client.rs`) rewrites `Ultra` to `Max` unconditionally before the request is sent, so the API never receives it.
- This wrapper mirrors that: `ultra` stays selectable in configuration and on model ids, and is emitted as `max`. The `WireReasoningEffort` type excludes `ultra` outright, so a future change cannot reintroduce it on the request path without a type error.

## Request Routing

### Fixes

- **Codex Max is no longer rerouted by the new `max` effort suffix.** `gpt-5.1-codex-max` is a model id that happens to end in `-max`, not Codex at `max` effort. Adding `max` to the effort-suffix pattern without care would have stripped the suffix and resolved every Codex Max request onto `gpt-5.1-codex`. The pattern now scopes a lookbehind to the `max` alternative only, so `gpt-5-codex-low` still parses `low` while `gpt-5.1-codex-max` is left intact.
- **Unrecognised `gpt-5.6-*` identifiers no longer silently resolve to `gpt-5.5`.** The general GPT-5 resolver has no minor `6`, so anything it did not match fell through to the stable variant — meaning a `gpt-5.6-sol` request ran GPT-5.5 instead. A dedicated GPT-5.6 resolver now claims those identifiers, mapping unrecognised tiers to Sol.

## Internals

- The reasoning-effort union and the model-id suffix pattern are both derived from a single ordered `REASONING_EFFORTS` list in the leaf `lib/constants.ts`, so a future tier cannot be added to one and forgotten in the other.
- `ModelReasoningEffort` and `WireReasoningEffort` moved from `lib/request/helpers/model-map.ts` into `lib/constants.ts`, keeping the base types layer out of the request layer. `lib/schemas.ts` imports model-map and `lib/types.ts` imports schemas, so defining the union in `lib/types.ts` would have closed an import cycle. Model-map re-exports both types, so every existing caller is unchanged.

## Testing

### Improvements

- Added GPT-5.6 coverage for tier resolution, the flagship alias, per-tier effort aliases, the absence of `none`/`minimal`/Luna-`ultra` aliases, per-tier defaults, the `ultra` → `max` wire rewrite, `max`/`ultra` step-down on pre-5.6 models, pricing, and the two routing fixes above.
- The suffix-pattern test is driven from `REASONING_EFFORTS` itself, so a tier added to the union but missed by the pattern fails the suite. A snapshot of the pattern source would not catch that, since a re-hardcoded list matching today's tiers would still pass.
- Added `test/module-boundaries.test.ts` covering the `lib/types.ts` → `lib/request/**` import boundary directly. `import-x/no-cycle` does not catch a regression there, because model-map no longer depends on the types layer, so re-pointing the import would introduce no cycle.

## Docs

- Documented the tiers, the `max`/`ultra` levels, the `ultra` → `max` behavior, the `none`/`minimal` coercion, and the `gpt-5.6` → Sol alias in the configuration reference and the config template README.

## Notes

- Minor release published under the `latest` dist-tag (`npm i -g codex-multi-auth`).
- Pricing recorded for cost estimation: Sol `$5` in / `$30` out, Terra `$2.50` in / `$15` out, Luna `$1` in / `$6` out, per 1M tokens, with cached input at 10% of input.
- GPT-5.6 has no unsupported-model fallback chain into `gpt-5.5`. An account without GPT-5.6 access fails the request rather than degrading. The default model and `QUOTA_PROBE_MODEL_CHAIN` are deliberately untouched.
- GPT-5.6 routes to the `gpt-5.2` prompt family. Upstream ships no `gpt_5_6_prompt.md`; the 5.6 base instructions live inline in the model catalog, and `gpt-5.5` already reuses that family the same way.
- `.codex-plugin/plugin.json` had drifted to `2.3.3` and is realigned to the package version in this release.
