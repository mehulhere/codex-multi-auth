# v2.4.0

Adds per-invocation account forcing for the `codex-multi-auth-codex` wrapper. This is an additive feature release: when the new flag and environment variable are unused, routing, account selection, storage layout, and the normal auth flow are unchanged.

## Wrapper

### Features

- New `codex-multi-auth-codex --account <index|email|id>` forces a single forwarded Codex invocation onto one account. The selector accepts a 1-based index (matching `codex-multi-auth list`), an email, or an account id; an all-digit selector is always treated as an index. `CODEX_MULTI_AUTH_FORCE_ACCOUNT=<selector>` has the same effect for tools that set environment variables more easily than flags, and the `--account` flag wins when both are present (#623, #624).
- The pin is **ephemeral**: it is applied only by that invocation's own runtime-rotation-proxy instance and never mutates the persisted `switch` pin on disk, so concurrent invocations with different `--account` values do not interfere.
- The pin is **fail-hard**: the session never rotates, and if the chosen account is rate-limited or otherwise unavailable the request fails (reusing the existing `codex_pinned_account_unavailable` path) rather than silently using a different account. If the runtime rotation proxy is disabled (or `CODEX_MULTI_AUTH_BYPASS=1`), or the selector does not resolve to a configured account, the wrapper exits non-zero without launching Codex.

## Runtime Rotation

### Internals

- The wrapper resolves the selector to a 0-based index against the same scoped accounts pool the proxy loads, strips the launcher-only `--account` flag from the arguments forwarded to the official Codex CLI, and publishes an internal `CODEX_MULTI_AUTH_FORCE_ACCOUNT_INDEX` so the value survives the launcher → detached app-helper process boundary. The proxy consumes it as an ephemeral pin (`pinnedIndex = forcedAccountIndex ?? persistedPinnedIndex`), reusing the existing deterministic-pin selection path so no other routing behavior changes. The forced-pin variable is scrubbed from unrelated background/management children so it can never leak beyond the single forwarded run.

## Testing

### Improvements

- Added coverage for the whole path: proxy-level deterministic selection to the forced account, fail-hard with no upstream call when unavailable, the environment-variable consumption branch and its precedence versus an explicit option (including a forced index of `0`); and launcher-level resolution by index, email, account id, `--account=` form, and `CODEX_MULTI_AUTH_FORCE_ACCOUNT`, flag-over-env precedence, the rotation-disabled and out-of-range error paths, argument stripping, and propagation of the resolved index into the detached app-helper process.

## Docs

- Documented the feature in the command reference (a dedicated "Force an account for one invocation" section and a Common Flags entry), the configuration reference (the `CODEX_MULTI_AUTH_FORCE_ACCOUNT` override and the internal index variable), the feature map, and the README.

## Notes

- Minor release published under the `latest` dist-tag (`npm i -g codex-multi-auth`).
- Design note: the selector is resolved through a single seam so a future `--account <group>` (restricting rotation to a named subset of accounts) can layer on without reworking today's single-account pin.
